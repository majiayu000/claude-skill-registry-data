# 문제 가설 정의

## 목적

고객 여정 분석과 인터뷰 결과를 기반으로 핵심 문제를 테스트 가능한 가설로 정의합니다.

## 사용 시점

- 고객 여정 맵 작성이 완료된 후
- 킹핀 문제 정의 전
- 해결해야 할 핵심 문제를 명확히 해야 할 때
- 사용자가 "문제 가설", "문제 정의", "핵심 문제"를 언급할 때

## 필수 입력
- 타겟 고객 정의: `define/고객분석.md` (customer-analysis 결과)
- User Journey Map: `define/유저저니맵.md`, `define/유저저니맵.svg` (journey-mapping 결과)
- 고객 경험 조사 데이터 (customer-experience 결과):
  - `define/고객경험인터뷰결과.md`
  - `define/고객경험인터뷰결과취합.md`
- 고객경험 단계: `define/고객경험단계.md` (선택사항)

## 문제 가설 프레임워크

### 1. 문제 가설

#### 주요 문제 가설
**[타겟 고객]**은(는) **[상황/맥락]**에서 **[핵심 문제]**를 경험하며, 이는 **[부정적 결과]**를 초래합니다.

**예시:**
"바쁜 직장인들은 점심시간에 건강하고 맛있는 식사를 찾기 어려워하며, 선택 과정에서 많은 시간을 낭비하고, 이는 불만족스러운 식사 또는 점심 결식으로 이어집니다."

#### 보조 문제 가설 (2-3개)
관련 부차적 문제들을 나열합니다

### 2. 근본 원인 분석

"5 WHY" 기법 적용:
- **Why #1**: {첫 번째 이유}
- **Why #2**: {두 번째 이유}
- **Why #3**: {세 번째 이유}
- **Why #4**: {네 번째 이유}
- **Why #5**: {근본 원인}

### 3. 문제 영향 범위

문제를 정량화합니다:
- **빈도**: 얼마나 자주 발생하는가?
- **심각도**: 고객에게 얼마나 중요한가?
- **영향받는 고객 수**: 몇 명이 이 문제를 경험하는가?
- **비용/시간 손실**: 정량적 영향은?

### 4. 현재 대안 및 한계

기존 솔루션 분석:
- 고객이 사용하는 현재 workaround
- 각 대안의 한계점
- 기존 솔루션이 불충분한 이유

### 5. 테스트 가능한 가설 진술

검증을 위한 가설 형식화:

"우리는 **[타겟 고객]**이(가) **[특정 문제]**를 가지고 있다고 믿습니다. 만약 우리가 **[솔루션/제품]**을 제공한다면, **[측정 가능한 결과]**가 발생할 것입니다."

### 6. 검증 계획

가설을 테스트하는 방법 정의:
- 검증을 위한 핵심 질문 3-5개
- 측정 지표 (KPI)
- 검증 방법 (인터뷰, 설문, 테스트)
- 가설이 참인지 판단하는 기준

## 중요 가이드라인

- 문제는 구체적이고 테스트 가능해야 함
- 문제 자체에 집중하고, 솔루션은 제외
- 구체적이고 관찰 가능한 증거 사용
- 가정 기반 진술 회피
- 가설은 반증 가능해야 함
- 명확한 성공 기준 정의
- 문제와 솔루션을 분리

## 도구 활용

### Sequential MCP 사용
근본 원인 분석과 가설 검증 계획 수립 시 Sequential MCP를 활용하여 체계적으로 문제를 분석하세요.

## 주의사항

- 문제는 고객 관점에서 작성
- 솔루션은 포함하지 않음
- 측정 가능한 지표 정의
- 5 WHY는 최소 3단계 이상
- 현재 대안 분석 필수
- 검증 가능한 가설로 작성
- 정량적 데이터 포함
